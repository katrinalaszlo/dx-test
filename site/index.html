How it works

1

Point it at your product

dx-test reads your routes, docs, and config. Express, Spring Boot, Go, Python — it finds the endpoints automatically.

2

It generates test flows

Happy paths and error paths derived from your actual API. Auth flows, CRUD operations, validation checks — all generated.

3

It walks your app like a real user

Opens a real browser. Hits real APIs. Navigates every page. Captures console errors, network failures, broken error states, and screenshots.

4

You get a bug report, not a test report

What the user was doing, what broke, the network error, a screenshot. Everything a dev needs to fix it without asking questions.

Why this exists

Every testing tool checks whether your code works. Nobody checks whether someone can successfully use your product.

dx-test does what a perfectionist PM does manually — walks the app, inspects the experience, catches what automated tests miss.

Built with the Divergent framework

dx-test follows the principle that most things should use defaults — and the few places that matter should get all the attention.

  • Tests experience, not code — bugs are "user got confused" not "assertion failed"
  • Bug reports, not test results — actionable tickets with full context
  • Real apps, not mocks — real browser, real API calls, real pages
  • PM workflow — walk, find, capture, fix, re-walk
